### 修改 1：`weighted_s` 回退到 `shift()`（`data_function.py`）

#### 背景
5 月 13 日的修复把 `weighted_s` 从 `shift()` 改成了 `reindex(index - 1min) + ffill`，原因是实时 `pred_df` 索引不连续，`shift(1)` 会按物理位置下移导致错位（比如 21:30 的 weighted_s 错用了 21:28 的值）。

#### 今天的问题
用户反馈：`reindex + ffill` 会**掩盖数据缺失**。如果某个分钟真的丢了，`ffill` 会静默用前值填充，导致实时和研究表面一致，但实际数据质量已下降。用户要求**暴露问题而不是掩盖问题**——如果丢分钟，diff 应该变大，这样才知道管道有问题。

#### 具体改动
**文件**：`/home/strategy_PAMY_dev/data_function.py`，函数 `run_345()`

**改回前**（5 月 13 日版本）：
```python
w = pred_df['weighted']
w_prev = w.reindex(w.index - pd.Timedelta(minutes=1))
w_prev2 = w.reindex(w.index - pd.Timedelta(minutes=2))
df['weighted_s'] = w * 0.6 + w_prev * 0.3 + w_prev2 * 0.1
```

**改回后**（今天版本）：

#### 依赖条件
- 这要求 `pred_df` 索引**严格连续**（每分钟一行）。

### 修改 2：DataService 增加分钟 gap filling（`data_service.py`）

#### 背景
实时行情可能因网络抖动、ZMQ 丢包、数据源本身缺数据等原因，导致某些分钟没有 tick 到达。`DataService` 原来检测到新分钟时直接推送，如果中间跳过 N 分钟，`pred_df` 索引就会断裂，下游所有 `shift()`、`pct_change()`、`rolling()` 都会错位。

#### 具体改动
**文件**：`/home/strategy_PAMY_dev/data_service.py`

在 `run()` 主循环的 `_get_latest_datetime()` 之后，新增 gap detection 逻辑：

```python
missing_minutes = int((latest_time - time_before).total_seconds() // 60) - 1
if missing_minutes >= 1:
    fill_count = 0
    for i in range(1, missing_minutes + 1):
        if fill_count >= MAX_FILL:
            break
        fill_time = time_before + pd.Timedelta(minutes=i)
        if not self._is_in_trade_session(fill_time):
            continue
        # 构造 fill_package 并放入 queue
        fill_count += 1
```

**关键参数**：
- `MAX_FILL = 5`：最多补 5 分钟。超过 5 分钟认为是大面积断连，不再补点，避免用陈旧数据污染。
- `_is_in_trade_session(dt)`：补点前过滤非交易时段，避免在休盘期间生成假分钟。

#### `_is_in_trade_session()` 实现
```python
def _is_in_trade_session(self, dt: pd.Timestamp) -> bool:
    t = dt.time
    if datetime.time(11, 31) <= t < datetime.time(13, 30): return False   # 午休
    if datetime.time(10, 15) <= t < datetime.time(10, 30): return False   # 小休
    if t >= datetime.time(15, 0) and t < datetime.time(21, 0): return False  # 日盘结束到夜盘开始
    if t >= datetime.time(23, 0) or t < datetime.time(9, 0): return False    # 夜盘结束到日盘开始
    return True
```

**注意**：这里没有处理 11:30 本身（11:30 是合法交易时间，11:31 开始才是午休），也没有处理 10:15 之前和 10:30 之后。

#### 验证方式
数据源恢复后，观察 `save_files/{SYMBOL}/predictions/predictions_*.csv` 的时间列，检查是否存在连续分钟缺行。如果缺行，说明 gap filling 没生效或 `MAX_FILL` 不够；如果有 11:31-13:29 之间的行，说明 `_is_in_trade_session` 没过滤成功。

#### 背景
旧版 `_save_min_data` 把 `data_dict[f'{main}_tick']` 的**全部历史 tick** 都保存到 CSV。随着运行时间增长，文件体积膨胀，且每次触发都全量重写，IO 开销大。此外，旧 CSV 没有时间戳字段，无法知道这行数据是策略在哪一分钟触发的。

**文件**：`/home/strategy_PAMY_dev/strategies.py`

**改前**（保存全部历史）：

**改后**（只保存当前分钟 + 新增字段）：
```python
def _save_min_data(self, data_dict, time_recently, readable_time):
    main = self.main_symbol
    tick_df = data_dict.get(f'{main}_tick', pd.DataFrame())
    current_minute = pd.to_datetime(time_recently).replace(second=0, microsecond=0)
    next_minute = current_minute + pd.Timedelta(minutes=1)
    mask = (tick_df['datetime'] >= current_minute) & (tick_df['datetime'] < next_minute)
    current_tick = tick_df[mask].copy()
    
    current_tick['trigger_time'] = readable_time          # 策略触发时间（字符串）
    current_tick['last_tick_time'] = current_tick['datetime'].iloc[-1]  # 该分钟最后一条 tick 时间
    
    sec5 = current_minute + pd.Timedelta(seconds=5)
    post_sec5 = current_tick[current_tick['datetime'] >= sec5]
    avg_price = post_sec5['last_price'].mean() if not post_sec5.empty else current_tick['last_price'].mean()
    current_tick['avg_price_from_5s'] = avg_price         # 第5秒后的均价，用于延迟成交定价
    
    current_tick.to_csv(fpath, index=False)
```

**字段说明**：
- `trigger_time`：策略逻辑触发时的时间字符串，用于追溯
- `last_tick_time`：该分钟内最后一条 tick 的时间，可用来判断该分钟数据是否完整
- `avg_price_from_5s`：当前分钟第 5 秒及之后的 `last_price` 均值。如果该分钟前 5 秒内无 tick（如整点边界），fallback 为全分钟均值

**保存路径**：`save_files/data/{symbol}/{date}/{symbol}_{timestamp}.csv`

## 已知陷阱（别踩）

- **test 环境 ZMQ 端点**：`172.17.0.6:7779`（模拟数据源），与 prod `192.168.2.238:77718` 不同。旧脚本 `接收生产levle2行情.py` 连的是 prod，不能直接用来测试本地环境。
- **sql_writer 无独立日志文件**：旧进程 stdout 挂在 pts/6，关闭终端会丢输出。重启时务必重定向到 `logs/sql_writer_YYYYMMDD.log`。
- **`shift()` 在缺失分钟时会错位**：`weighted_s` 已回退到 `shift()`，如果 data_service 的 gap filling 没补上全部空缺，实时值会与 research 错位。这是**已知取舍**——先保证数据完整性（不掩盖异常），而不是靠计算层容错。
- **monitor_web 延迟成交逻辑**：它读取的是 `save_files/data/` 里的 CSV，而不是直接读 DB。如果 `strategies.py` 的 `_save_min_data` 字段名改了，monitor_web 的解析会崩。
- **prod auto-archive**：`finally` 块只在正常退出（15:00 auto-exit 或 Ctrl+C）时触发。如果进程被 `kill -9`，归档不会执行。
- **Trade-date 规则**：夜盘 21:00-23:00 归属**下一个** trade_date。缓存目录和归档目录都受此影响。
- **gap filling 的 `_is_in_trade_session`**：目前只过滤了 11:31-13:30、10:15-10:30、15:00-21:00、23:00-09:00。如果数据源在非交易时段推送了数据，gap filling 不会补这些时段（这是对的），但如果数据源在交易时段边界（如 09:00:01）才有第一条数据，08:59 到 09:00 之间的 gap 不会被补（因为 08:59 不在交易时段内）。

## 修正后的已知陷阱

- **`data_service.py` 不要加 `_is_in_trade_session`**：prod 和 dev 现在都是简单判断 `minute != minute`，策略端自己处理时段过滤。加了这个方法会导致 21:00-21:09、09:00-09:09 等时段不触发。

- **旧历史文件可以删**：`history_save_files/20260518/data/` 下的旧 `*_min_*.csv` 是旧逻辑产物，每个都包含从 3月26日至今的全部分钟线，重复了 5000 多次。
