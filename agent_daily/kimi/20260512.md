- **实时优化**：当 `len(tick_data) < 5000` 时，`valid_index` 被截断为**最后一个点**（只算最新 bar）

### 排查 2：为什么 `A_P_vcorr10` 等 cross 因子 realtime = nan？

**排查过程**：
1. 用户怀疑 `dict_keys` 和 `other_mins` 顺序传错 → 检查代码后发现顺序一致
2. 独立脚本验证：`A_P_vcorr10` 计算结果 = 0.935（正确），但 save_files 中 = nan
3. 发现关键代码（`data_function.py:733`）：
```python
if len(self.tick_data) < 5000 and len(self.valid_index) > 1:
    self.valid_index = self.valid_index[-1:]
```

**根本原因**：
- 实时模式下 `valid_index` **只有 1 个点**（最后一根 bar）
- `vcorr10` 计算 `rolling(window=10).corr()`：1 行数据算不了 10 窗口 rolling → **nan**
- `volumediv_diff` 同理：1 行数据算不了 rolling sum → **nan**
- `closepctchg20_sub` 为什么有值？因为 `pct_change(20)` 是**先在完整历史数据上算好，再 reindex 取最后一个点**，所以能有值

**这不是 bug，是设计如此**。实时模式下只计算最新 bar，rolling 类 cross 因子在单点模式下天然为 nan。

## 容易判断错误的细节（极重要）

### 1. 交易日 vs 日历日期（最容易踩坑）
- **夜盘 21:00-23:00 归属下一个交易日**
- 例：03-23 21:00 的夜盘 → trade_date = 03-24
- `all_factor.feather` 中的 `datetime` 是**日历日期**，所以 03-24 的因子包含 03-24 白天盘 + 03-23 夜盘
- `check_intraday.ipynb` 现在用 `night_date` / `day_date` 手动指定，务必注意对应关系

### 2. 回放数据的实际内容 vs 文件名
- 文件名是 `2026/3/24/night`，不代表数据就是 03-24 的价格
- 必须通过**价格比对**确认回放的是哪一天（用 1min 数据对比 open/close）

### 3. 实时模式下 `valid_index` 只有一个点
- 这是 `Factor_generator` 的优化设计，不是 bug
- **后果**：`vcorr10`、`volumediv_diff`、`cvcorr10_diff` 等 rolling 类 cross 因子在单点模式下返回 **nan**
- `closepctchg_sub`、`oi5_diff` 等 `pct_change` 类因子不受影响（先在完整数据上算好再取点）
- **判断方法**：如果只有 cross 因子的 rolling 类为 nan，而 pct_change 类有值 → 正常，不是顺序传错

### 4. `_save_min_data` 存的是完整历史 + 实时数据
- 不是只存最新 bar，是存完整 `min_cache`
- 历史数据可能和实时数据之间**有断层**（如 03-23 15:00 后直接跳到 05-12 21:01）
- Market Check 时只对比**最新日期**的部分，避免历史数据干扰

### 5. LH 等无夜盘品种的数据在夜盘不为空
- `min_cache` 中有历史 CSV 数据，所以 LH 的 min_data 在夜盘时段不为空
- `P_LH_*` 因子在夜盘能算出值，但用的是 LH 的历史白天盘数据，需注意物理意义
