## 一、生产方向

### 1. 架构升级：数据本地化处理

将旧版策略系统升级为新版架构，核心目标是**行情数据本地处理，降低交易机负载**。

| 项目 | 旧版 | 新版 |
|------|------|------|
| 运行位置 | 交易机直接运行 | 行情拉取到本地处理 |
| 策略脚本 | 4个独立入口 (A/M/P/Y) | 统一调度 (orchestrator.py) |
| 并行方式 | ThreadPoolExecutor（受GIL限制） | ProcessPoolExecutor (spawn模式，真正多核) |
| 数据管道 | 策略脚本内嵌 | 独立 DataService + sql_writer_dce.py |
| 因子缓存 | 预测值缓存 | 因子缓存（可重建，更安全） |

**主要工作**：
- 将策略调度从 ThreadPoolExecutor 改为 ProcessPoolExecutor (spawn模式)，解决 GIL 限制下的并行效率问题
- 修复子进程漂移问题，改为主进程统一维护状态
- DataService 和 sql_writer_dce.py 拆分为独立模块，行情写入与策略计算解耦
- 因子缓存从预测值缓存改为因子缓存，避免因计算顺序导致的数据丢失

**性能提升**：8策略并行总耗时 30s → 2s。

### 4. 行情监控页面 `monitor_web.py`（新增）

**设计目标**：把交易监控/回放逻辑从 `strategies.py` 中彻底剥离，做成独立进程。

**解决的问题**：
- 策略代码臃肿，监控逻辑与交易逻辑耦合
- 监控逻辑修改需要重启 orchestrator
- 监控 bug 可能影响策略主循环

**架构**：
- Flask Web 服务，端口 5000
- 后台线程每 5 秒扫描一次 `save_files/`
- 完全无状态：不读 SQLite，只读 `strategies.py` 生成的 CSV 文件

**核心功能**：
- **启动时 `replay_history()`**：遍历 `save_files/` 下所有已存在的 `predictions_*.csv` 和 `data/` 下的 tick CSV，按时间顺序重建历史状态。即使 monitor_web 中途重启，也不会丢历史交易记录。
- **延迟 1 分钟成交机制**：策略在分钟 T 产生信号，monitor_web 不立即记录成交，而是等到分钟 T+1 的数据到达后，用 T+1 的 `avg_price_from_5s` 作为成交价。模拟实际交易中"信号产生后下一分钟才能执行"的延迟。
- **实时盈亏统计**：每个品种输出 `trades.json`，Web UI 展示品种状态、今日交易列表、累计盈亏（按点数）。

**路径**：`/home/strategy_PAMY_dev/monitor_web.py`

## 二、研究方向

### 1. 损失函数差异化配置实验（8品种 × 8方案）

针对 DCE 农产品 8 个品种（A/B/C/CS/M/Y/P/LH），批量对比了 8 种训练方案：

| 方案 | 说明 |
|------|------|
| baseline | MSE 基础版 |
| baseline_smooth | MSE + 标签平滑 |
| trick5_huber | Huber 损失 |
| trick5_huber_smooth | Huber + 标签平滑 |
| trick6_direction | 方向敏感损失 |
| trick6_direction_smooth | 方向敏感 + 标签平滑 |
| trick8_quantile_mse | MSE + 分位数样本权重 |
| trick8_quantile_huber | Huber + 分位数样本权重 |

**核心结论**：
- **C / LH**：MSE 最优（test_corr: 0.0544 / 0.0621）
- **Y / M / A / P / CS / B**：Huber 最优
- **标签平滑全面翻车**：平均下降 -0.026，推理端已做平滑，训练端重复做 = 二次平滑
- **方向敏感损失效果平庸**：avg corr 0.017，维护成本高
- **分位数样本权重**：A/B/CS/LH 报 NaN，有值品种提升微弱

**已记录**：`/home/notes/model/2026-05-29_experiment_tricks_8方案对比结论.md`

## 三、遇到的问题及解决

| 问题 | 根因 | 解决方案 |
|------|------|---------|
| 回测收益率异常低（~0.5%） | multi_symbol_eval 使用默认 1000 万本金，原脚本用动态本金 | 新增 `money='auto'` 模式，按 margin/0.4 计算 |
| P 品种 `RPP_5D` 差异 | DataService `min_cache` 被 `aggregate_ticks` + `update_concat` 覆盖 | 部署 21:15 快照，分层保存数据排查 |
| 标签平滑效果差 | 推理端已做平滑，训练端重复做 = 二次平滑 | **放弃标签平滑 trick** |
| 分位数权重 A/B/CS/LH 报 NaN | 可能与标签分布极端有关，分位数计算溢出 | **放弃分位数权重 trick**，后续排查根因 |
| 旧 notebook 散落各文件夹 | 训练流程分散在 20+ 个 ipynb 中，难以维护 | 统一重构为 `.py` 脚本 + 按日期分文件夹 |
