## 容易产生判断错误的地方（重点）

1. **不要假设 logger 和 save_files 的数据来源一致**
   - 从代码上看，logger 保存的 `fac_generator.min_data` 和 `_save_min_data` 用的 `data_dict['P_min_data_concat']` 理论上是同一个 DataFrame 引用。
   - **但实际值不同**（close 9724 vs 9710，bar_count 25 vs 237）。这意味着某个环节改变了对象内容或者它们并非真正的同一对象。此前的 debug 不要基于“应该是同一个”做推断。

2. **不要混淆数据的三个层级**
   - **层级 A** ：SQLite 里的原始 tick（sql_writer 写入）→ 通常是正确的。
   - **层级 B** ：DataService 的 `tick_cache` / `min_cache`（`read_table` + `aggregate_ticks` + `update_concat`）→ 这里可能被覆盖、截断或污染。
   - **层级 C** ：策略层接收到的 `data_package` / `fac_generator` → 这里的值和 B 层可能已经不一致。
   - 今天的快照就是为了同时抓住 B 层和 C 层，做对比。

3. **`aggregate_ticks(df)` + `update_concat(min_cache, new_min)` 是高危操作**
   - `_refresh_instrument` 每次都会用 `aggregate_ticks(df)` 重新聚合分钟线，然后 `update_concat` 用 `drop_duplicates(keep='last')` 覆盖旧数据。
   - 如果 `df` （即 `read_table` 读出的 tick）不完整，聚合出来的 min bar 就是错的，但仍然会覆盖掉正确的历史 min。
   - **误区**：以为 `update_concat` 只是追加新行，实际上它会用新聚合的值覆盖同一时间戳的旧值。

4. **WAL 模式 + 连接缓存 = 数据不同步风险**
   - sql_writer 这周切到了 WAL 模式。
   - data_service 使用 `get_db_connection` 全局缓存连接。
   - 在 WAL 模式下，缓存连接可能看不到最新写入的数据，导致 `read_table` 返回的 tick 是旧版本。
   - **误区**：以为 SQLite 里有完整数据，DataService 读出来的就一定是完整的。

## 更新（10:49）

### 根因定位 ✅

**不是 `aggregate_ticks` 覆盖问题，也不是 WAL 问题。**

真正的原因是 `data_service.py` 第284行 `_package_data(raw_time)` 传了**含秒的时间**：

- DB 最新 tick 时间 = `21:15:23`
- `raw_time = 21:15:23`（含秒）传给 `_package_data`
- `tick = df[df.datetime <= 21:15:23]` 把 `21:15:00.250 ~ 21:15:02` 的 tick 也打包进去了
- 这些 tick 被 `Factor_generator.resample('1min', label='right', closed='right')` 归入了 **21:16:00** 的分钟桶
- `valid_index` 多出一个 21:16:00，实时优化截断后只剩 `[21:16:00]`
- **结果：21:15:00 这一帧算的是 21:16:00 的因子值**，和研究环境的 21:15:00 对不上
- 这也解释了为什么 `first` 类因子碰巧能对（21:16:00 桶的 first 就是 21:15:00.250 的价格），而 `mean/last/vwap/rolling` 类因子全错

### 修复 ✅
